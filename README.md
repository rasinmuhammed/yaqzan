In August 2018, the Kuttanad region of Kerala experienced its worst flooding in a century. The emergency operations centers did not fail because of a lack of data. They failed because human operators could not process the massive influx of desperate WhatsApp messages, broken telemetry, and conflicting reports fast enough. 

Dashboards can show you that a hospital generator is failing. They cannot tell you which of the fifty stranded rescue boats to reroute to save the ICU patients in the next ten minutes.

Yaqzan (Arabic for "The Vigilant One") solves the cognitive bottleneck in disaster response. It is a live war room simulation where an AI model does not just route data, but actively reasons through it. 

We built Yaqzan to prove that when you connect a reasoning model (K2 Think V2) to a deterministic simulation engine, you get an Incident Commander capable of continuous, high-speed tactical triage.

## Core Architecture

Yaqzan is designed around a three-part closed loop:

**1. The Simulation Engine**
Running locally in Python, the engine simulates a flooding event in Kuttanad. It models physical phenomena on a graph dataset. The graph consists of `Nodes` (towns, hospitals, pump stations) and `Edges` (roads). Every two seconds, the simulation ticks. Water levels rise across nodes, edge passability drops, cell towers go offline, and predefined scenario injects (like citizen reports or dam breaches) are fired into the system.

**2. The K2 AI Commander**
Instead of hitting the model with unstructured text, we compile the entire city state into a compact JSON payload. K2 ingests this state, looks at incoming citizen reports, and generates a live reasoning trace. You can watch it think on screen: identifying trapped citizens, checking water depths, verifying hospital capacities, and drafting rescue operations.

**3. The Ground-Truth Verifier**
Safety is non-negotiable. Before any AI directive reaches the map, it hits our Verifier. The AI model operates under a strict "Closed World" assumption. If the AI hallucinates a rescue boat that does not exist in the state payload, or tries to route a truck through an impassable road, the Verifier explicitly rejects the command. It forces the AI to correct its mistake in the next cycle.

## Technical Implementation Details

### The Backend (FastAPI + Asyncio)
The backend is responsible for maintaining the simulation state and handling the WebSocket streams.
*   **Engine (`app.sim.engine`)**: The deterministic physics model. It calculates water depth changes, infrastructure failure probabilities, and unit movement vectors.
*   **Commander Loop (`app.commander`)**: An asynchronous loop that runs parallel to the simulation. Every `N` ticks, it serializes the `StateSnapshot`, sends it to the K2 LLM, and waits for the Command Plan response.
*   **WebSockets (`app.main`)**: Broadcasts the `StateSnapshot` and any AI reasoning chunks to all connected frontend clients at 60Hz.

### The Frontend (React + Vite)
The frontend is a React application built to mimic a high-stakes emergency operations center.
*   **Live AI Triage (`ReportsDash.tsx`)**: When a citizen submits an incident report via the public portal, it appears instantly in the Reports Dashboard. The AI streams its tactical deliberation live, stripping away conversational filler and focusing entirely on unit dispatch and survival instructions.
*   **Ask Commander AI (`ChatPanel.tsx`)**: Operators can ask the AI complex strategic questions. For example: "We have one boat left, but two critical emergencies. Who do we rescue?" The AI will cite actual water depths and unit positions to justify its tradeoff.
*   **Map Visualization (`MapPanel.tsx`)**: A custom SVG rendering engine that plots the graph nodes, draws the road edges, and animates unit movements in real time based on the WebSocket payloads.

### Human-in-the-Loop Operations
Yaqzan advises, it never commands. Every rescue operation generated by the AI sits in a queue waiting for human approval. The system generates a `RescueOp` JSON object containing the target node, the assigned unit, and the priority level. The operator must explicitly click "Approve" before the unit actually moves on the map.

## Modding and Customization

### Creating Scenarios
You can create entirely new disaster scenarios by modifying the JSON files in `backend/app/sim/scenarios`. A scenario file defines:
*   The baseline graph (nodes, edges, facilities).
*   The available response units (boats, helicopters, trucks).
*   The `injects` array, which specifies exact text events and severity levels to trigger at specific simulation ticks.

### Prompt Engineering
The system prompts are located in `backend/app/commander/prompts.py`. If you want to change how the Triage AI behaves or how strictly the Commander AI responds to chats, you can edit these files. The prompts are highly sensitive and require explicit negative constraints (e.g., forbidding the AI from outputting XML tags).

## Setup & Deployment

You can run the entire simulation locally.

**1. Start the Backend**
The backend requires Python 3.11+.

```bash
cd backend
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp ../.env.example .env
```

If you have K2 credentials, add them to your `.env` file. If you do not, the system will automatically fall back to a deterministic "scripted" commander so you can still run the demo locally without internet access.

```bash
uvicorn app.main:app --port 8000
```

**2. Start the Frontend**
The frontend is built with React and TailwindCSS.

```bash
cd frontend
npm install
npm run dev
```

The War Room will be available at `http://localhost:5173`. 

## Demo Pitch Mode

If you are presenting Yaqzan to a panel of judges, append `?film=true` to your localhost URL. This enables a cinematic presentation mode. It hides development buttons and allows you to trigger massive crisis injects on cue by pressing the number keys (1, 2, 3) on your keyboard. 

## Testing

The simulation is fully deterministic. You can run the test suite to verify flood propagation, verifier rejections, and round-trip JSON parsing.

```bash
cd backend
python -m pytest
```
